## CONCEPT 1 ‚Äî What is SQLite?

SQLite is a lightweight, file-based database that runs entirely inside your app. No separate server. No internet connection needed. The entire database is stored as a single `.db` file on the device.

```
Without SQLite:
App data lives in memory ‚Üí lost when app closes

With SQLite:
App data written to habit_tracker.db file ‚Üí persists forever
```

This is why it is called **local-first** ‚Äî data lives on the device. In V2, this same local database syncs to the cloud. The local DB always remains the source of truth.

---

## CONCEPT 2 ‚Äî What is Drift?

Drift is a Flutter library that sits on top of SQLite. Without Drift, you write raw SQL manually:

```sql
CREATE TABLE habits (id INTEGER PRIMARY KEY, name TEXT NOT NULL);
INSERT INTO habits (name) VALUES ('Morning Walk');
SELECT * FROM habits;
```

This is error-prone, has no type safety, and requires manual mapping of results back to Dart objects.

**Drift lets you write Dart instead:**

```dart
class Habits extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get name => text()();
}
```

Drift reads your Dart classes, generates all the SQL, and returns results as typed Dart objects automatically.

```
You write:          app_database.dart      ‚Üê your configuration
You write:          habit_table.dart       ‚Üê your table definitions (Day 3)
build_runner generates: app_database.g.dart ‚Üê all boilerplate SQL code
```

You never write or edit `app_database.g.dart`. It is fully automated.

The `part 'app_database.g.dart'` directive at the top of `app_database.dart` tells Dart: "this file and the generated file are two halves of the same unit ‚Äî merge them together."

Tells Dart that `app_database.g.dart` is a part of this file. Both files together form one complete unit. The generated file contains the `_$AppDatabase` class that your `AppDatabase` extends.

```dart

```
Your class extends `_$AppDatabase` ‚Äî the generated parent class inside `app_database.g.dart`. The generated class contains all the internal Drift logic. You provide the configuration. Drift combines both.

---

```dart
AppDatabase() : super(_openConnection());
```
This is the constructor with a **parent constructor call**.

Understanding `super()` with a simple example:

```dart
class Parent {
  Parent(int value) {
    print("Parent received: $value");
  }
}

class Child extends Parent {
  Child() : super(10);  // must pass value to parent
}
```

Output: `Parent received: 10`

- `Child` extends `Parent`
- `Parent` constructor requires an `int`
- `Child` MUST call `super(10)` to give that value
- Parent constructor runs first

Now apply this to Drift:

```dart
// Generated parent class (inside app_database.g.dart)
abstract class _$AppDatabase extends GeneratedDatabase {
  _$AppDatabase(QueryExecutor e) : super(e);
}

// Your class
class AppDatabase extends _$AppDatabase {
  AppDatabase() : super(_openConnection());
}
```

- Parent `_$AppDatabase` requires a `QueryExecutor` (the database engine)
- `_openConnection()` returns a `QueryExecutor` (specifically a `LazyDatabase`)
- `super(_openConnection())` passes the engine to Drift
- Without this, Drift has no engine and cannot run any queries

---

```dart
@override
int get schemaVersion => 1;
```
Controls database migrations. If you later add new tables or columns, you increment this to `2`, `3`, etc. Drift detects the version change and runs your migration logic to update the existing database file without destroying user data. This is production-level database versioning.

## CONCEPT 5 ‚Äî `_openConnection()` Line by Line

```dart
LazyDatabase _openConnection() {
```
A top-level function (outside the class) that returns a `LazyDatabase`. It is outside the class because constructors in Dart cannot be `async`. The async work is wrapped inside `LazyDatabase` instead.

---

```dart
return LazyDatabase(() async {
```
`LazyDatabase` wraps the async database opening logic. It delays actually opening the SQLite file until the **first query runs**. The database is NOT opened when `AppDatabase()` is created ‚Äî only when the first actual query is executed.

Why? Because we cannot make a constructor `async` in Dart:

```dart
// WRONG ‚Äî not allowed in Dart
AppDatabase() async {
  final dir = await getApplicationDocumentsDirectory();
}

// CORRECT ‚Äî wrap async work in LazyDatabase
LazyDatabase(() async {
  final dir = await getApplicationDocumentsDirectory();
  return NativeDatabase(file);
});
```

---

```dart
final dir = await getApplicationDocumentsDirectory();
```
Gets the safe internal storage folder on the device:
`/data/user/0/com.yourapp.package/files/`

This is private to your app. Safe to store database files here.

---

```dart
final file = File(p.join(dir.path, 'habit_tracker.db'));
```
Creates the path to the SQLite file: `/data/user/.../files/habit_tracker.db`

`p.join()` safely combines the directory path with the filename using the correct path separator for the current platform.

This is the actual file where all your habits and completions are stored on disk. If the user uninstalls the app, this file is deleted.

---

```dart
return NativeDatabase(file);
```
Creates the SQLite engine pointing at that file. This tells Drift: "use the SQLite engine on this specific file." Drift now knows exactly where to read and write data.

## CONCEPT 6 ‚Äî When is SQLite Actually Opened?

This is the LazyDatabase timeline:

```
Step 1: AppDatabase() created         ‚Üí LazyDatabase object created, no file touched
Step 2: Riverpod provider created      ‚Üí AppDatabase() called, still no file opened
Step 3: HomeScreen watches provider    ‚Üí still no file opened
Step 4: First query executed           ‚Üí LazyDatabase NOW runs the async function
Step 5: getApplicationDocumentsDirectory() called
Step 6: File path created
Step 7: NativeDatabase(file) created   ‚Üí SQLite file opened HERE
Step 8: Query executes on the file
```

SQLite opens **only at first query**. Not at app startup. Not when the provider is created.

## CONCEPT 7 ‚Äî `database_provider.dart`

```dart
final databaseProvider = Provider<AppDatabase>((ref) {
  return AppDatabase();
});
```

- `Provider<AppDatabase>` ‚Äî a Riverpod provider that holds a single `AppDatabase` instance
- Returns `AppDatabase()` ‚Äî creates the database object (but does NOT open SQLite yet ‚Äî remember LazyDatabase)
- Riverpod ensures only **one instance** of `AppDatabase` exists in the entire app. Every screen that reads this provider gets the same instance. This is important ‚Äî you never want two separate connections to the same SQLite file.

## Interview Questions ‚Äî Drift & Database Setup

---

**Q: What is SQLite and why use it for a mobile app?**

> SQLite is a lightweight, serverless, file-based database that runs entirely inside the app. It requires no network connection and persists data across app sessions. For a local-first app it is ideal ‚Äî fast, reliable, and works offline. In V2 we sync this local database to the cloud, but it always remains the source of truth.

---

**Q: What is Drift and why use it over raw SQLite?**

> Drift is a type-safe Dart ORM for SQLite. Instead of writing raw SQL strings ‚Äî which are error-prone and have no compile-time checks ‚Äî you define tables as Dart classes and Drift generates all the SQL boilerplate via `build_runner`. Query results are returned as typed Dart objects. This eliminates an entire category of runtime errors and makes the data layer fully type-safe.

---

**Q: What is `LazyDatabase` and why is it used?**

> `LazyDatabase` is a Drift wrapper that delays opening the SQLite file until the first query is executed. It exists because Dart constructors cannot be `async`, but opening a database requires async file system operations like `getApplicationDocumentsDirectory()`. `LazyDatabase` wraps this async work cleanly. The result is better startup performance ‚Äî the database file is not touched until it is actually needed.

---

**Q: What is `schemaVersion` and why does it matter?**

> `schemaVersion` is an integer that tracks the structure version of the database. When you add new tables or columns, you increment it. Drift detects the version mismatch on app launch and runs your defined migration logic to update the existing database file. Without this, a schema change would either crash the app or silently ignore the new columns on existing devices.

---

**Q: Why is there only one `AppDatabase` instance in the app?**

> The Riverpod `Provider` creates the `AppDatabase` once and caches it. Every part of the app that reads `databaseProvider` gets the same instance. This is important because having multiple connections to the same SQLite file simultaneously can cause write conflicts and data corruption. A single shared instance ensures all queries go through one controlled connection.

---

**Q: What is `build_runner` and what does it generate for Drift?**

> `build_runner` is a Dart code generation tool. When you annotate a class with `@DriftDatabase` and run `dart run build_runner build`, Drift reads your table definitions and generates the `.g.dart` file containing all the SQL table classes, typed data classes for reading, companion classes for writing, and the query API. You never write or maintain this generated file manually.

---

**Q: What is the `part` directive in Dart?**

> The `part` directive splits one logical Dart file across multiple physical files. `part 'app_database.g.dart'` tells Dart that the generated file is part of the same library as `app_database.dart`. Both files share the same scope ‚Äî the generated `_$AppDatabase` class is accessible in `app_database.dart` even though it lives in a separate file. This is how Drift's code generation integrates seamlessly with your code.
